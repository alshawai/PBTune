# Security Policy

## Supported Versions

| Version | Supported          |
| ------- | ------------------ |
| 1.0.x   | :white_check_mark: |

## Reporting a Vulnerability

We take the security of PBTune seriously. If you discover a security vulnerability, please report it responsibly.

### How to Report

Email **contact.alshaw.ai@gmail.com** with a subject line starting with `[SECURITY]`.

Include the following information in your report:

- A description of the vulnerability
- Steps to reproduce the issue
- The potential impact of the vulnerability
- Any suggested fixes (if applicable)

### What to Report

- Authentication or authorization flaws
- Data exposure or leakage
- Injection vulnerabilities (SQL, command, etc.)
- Dependency vulnerabilities with a known exploit
- Configuration issues that could lead to compromise

### What NOT to Report

- Vulnerabilities in dependencies without a proof-of-concept exploit
- Issues that require physical access to a user's device
- Social engineering attacks
- Denial of service (DoS) attacks
- Issues already reported in public GitHub issues

### Expected Response Timeline

- **Acknowledgment**: Within **48 hours** of your report
- **Assessment**: Within **7 days**, we will provide an initial assessment of the report and an expected timeline for a fix

### Responsible Disclosure Policy

- Please give us a reasonable amount of time to address the issue before disclosing it publicly.
- We will coordinate with you on the disclosure timeline.
- We will credit reporters who follow responsible disclosure (unless you prefer to remain anonymous).
- We will not take legal action against researchers who report vulnerabilities responsibly and in good faith.

Thank you for helping keep PBTune and its users safe.
